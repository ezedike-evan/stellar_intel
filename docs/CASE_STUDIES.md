## Case study 1: $500 USDC → Lagos in 4 minutes

**Scenario.** A Nigerian freelancer wants to convert a $500 USDC payment
from an international client to NGN in their Kuda bank account. Date:
2026-03-18.

**Corridor.** USDC (mainnet) → NGN via bank transfer, beneficiary bank
code `058` (Kuda).

**Timeline.**

|       t+ | Event                                                                               |
| -------: | ----------------------------------------------------------------------------------- |
|    0.0 s | User opens `stellar-intel.vercel.app`, selects USDC → NGN, enters 500.              |
|    0.6 s | Rate table populated with three ranked anchors. Cowrie leads (net landed ₦703 500). |
|     38 s | User reviews TrustScore (84), confirms beneficiary, clicks Send.                    |
|     41 s | Freighter popup; user signs.                                                        |
|     43 s | Router relays the signed envelope to Cowrie; anchor returns interactive URL.        |
|     47 s | User redirected to Cowrie; completes SEP-24 KYC check (returning user, fast path).  |
| 3 m 20 s | Cowrie publishes `status: completed` + Stellar tx hash.                             |
| 3 m 58 s | NGN credited to Kuda account.                                                       |

**Net landed.** ₦703 500 (quoted), ₦703 500 (received). Zero drift.

## Case study 2: Agent-driven off-ramp via MCP

**Scenario.** Reproduce the flow from Case 1 using Claude Code + the
Stellar Intel MCP server; no human UI involved.

**Command trace** (edited for brevity):

```
>_ user: send $100 USDC to my NGN account 0123456789, bank code 058. pick the best rate.

>_ agent: calling stellar_intel.price_offramp(...)
       → 3 routes, best netLanded=141 230.00 via cowrie, trustScore=84.2
>_ agent: asking for confirmation (signer=freighter, deadline=9m58s)
>_ user: confirmed
>_ agent: calling stellar_intel.sign_intent(...)  → signature 64B ✓
>_ agent: calling stellar_intel.submit_intent(...)
       → interactiveUrl=https://cowrie.io/flow/sep24/…
>_ agent: handing URL to user
>_ user: completes KYC in new tab
>_ agent: polling stellar_intel.poll_intent(intentHash) every 15s
>_ agent: state=submitted → submitted → fulfilled
       → tx hash 0xabc… NGN 141 230.00 landed
```

**Wall-clock.** 3m 41s agent-active, plus 45s human time on KYC.

**Artifacts.**

- MCP transcript: `data/case-studies/case-02-mcp-transcript.txt`
- Agent log: `~/.local/share/stellarintel/agent-log.ndjson` snippet.
- No new key material introduced; signing went through the same
  Freighter the user already had.

**Point of the study.** Every flow available to a human is available
to an agent, and every safety rail the UI provides (deadline, review,
user attestation) the MCP server provides too.

## Case study 3: Multi-anchor split for $20k notional _(v2)_

_This case study is a **planned** release artefact and ships with v2._

**Scenario.** A $20 000 USDC → NGN remittance from a treasury operator.
At this notional the best single anchor quote is stale by the time the
intent signs; splitting across two anchors wins.

Expected outcome sketch (from sim dataset `data/sim/split-20k.json`):

- Single-anchor best: ₦28 121 500 net landed.
- Optimal 2-way split: ₦28 210 500 (Cowrie 60 %, Click 40 %).
- **Savings: +₦89 000 (+0.32 %)**.
- Split signing is a single user signature; the router fans the intent
  to both anchors, commits only after both return fresh quotes within
  the same 2 s window.

v2 scope includes capturing the real-money version of this study with
a sandbox counterparty.

## Case study 4: Anchor dispute and correction

**Scenario.** On 2026-02-11, Cowrie disputes a batch of four
`failed_observed` events for intents submitted between 08:12–08:17 UTC.
Cowrie's claim: a 5-minute Horizon outage, not an anchor fault.

**Resolution.**

1. Maintainer pulled Horizon's public incident log — confirmed the
   window (08:09–08:19 UTC, mainnet).
2. Added the window to `data/incidents.json` with reason
   `"horizon_outage"` and `excluded: true`.
3. For each of the four disputed observations, wrote a correction
   observation on-chain (`supersedes = <original hash>`, `excluded = true`).
4. Aggregates recomputed; Cowrie's `fill_rate` for the week restored to
   the pre-window value.

**Timeline.** Dispute opened 2026-02-11 15:04 UTC; corrections written
2026-02-13 10:22 UTC. Total resolution: < 48 hours, well inside the
7-day SLO.

**Artifacts.**

- GitHub issue: `#… (public)`
- Correction observation hashes: `data/case-studies/case-04-corrections.json`
- Recomputed TrustScore diff: `data/case-studies/case-04-score-diff.json`

**Point of the study.** The dispute process works, it is fast, and
corrections are a write rather than a delete — the original observation
remains in history for audit.

## How to capture a new case study

1. Pick a real flow (or a scripted one with testnet anchors).
2. Start the Playwright recording: `npm run capture:case -- --name case-XX-short-slug`.
3. Run the flow manually (or via the script at `scripts/benchmarks/flow-happy.ts`).
4. Commit:
   - Dataset at `data/case-studies/case-XX-*.json`.
   - Screenshots at `docs/showcase/images/case-XX-*.png`.
   - Write-up section added to this file using the template in
     `docs/templates/case-study.md`.
5. Open a PR titled `docs(case): add case-XX …`.

We favour **real, boring, reproducible** cases over dramatic ones. A
plain happy-path off-ramp is a better reviewer artefact than a heroic
recovery from an incident.
